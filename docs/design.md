# SonInk — AI 语音绘图工具 技术设计文档

> 一款纯语音控制的绘图工具：用户不使用鼠标和键盘，仅通过语音指令完成绘图创作。
>
> 本文档记录系统架构、核心模块设计、指令能力规划（计划支持 / 已实现 / 未完成及原因），并随开发进度持续更新。

- **状态**：设计阶段（随 PR 持续更新）
- **最后更新**：2026-06-12

---

## 1. 项目概述

### 1.1 设计目标

围绕题目的三个评分维度展开设计：

| 维度 | 设计应对 |
|---|---|
| 指令理解的准确性与容错性 | 同义词归一化 + 槽位提取 + 低置信度语音追问 + 全操作可撤销 |
| 语音到绘图操作的响应延迟 | 流式语音识别 + 双通道解析（简单指令走本地规则引擎，毫秒级响应） |
| 复杂指令的拆解与执行 | LLM 将复合指令拆解为原子操作序列（统一 DSL），顺序执行 |

### 1.2 核心原则

1. **解析与执行解耦**：所有指令解析结果统一为绘图 DSL（JSON 操作指令），规则引擎与 LLM 产出同一格式，执行引擎只认 DSL。
2. **快慢双通道**：高频简单指令本地解析（~50ms），复合指令才走 LLM（~1-2s），兼顾延迟与能力。
3. **闭环反馈**：用户没有鼠标悬停与点击确认，系统必须主动反馈——实时字幕（听到了什么）+ TTS 播报（做了什么）+ 画布高亮（选中了什么）。

## 2. 系统架构

### 2.1 总体流程

```
┌──────────┐   ┌─────────────────┐   ┌──────────────────────────┐
│  麦克风   │ → │ Web Speech API   │ → │       指令解析层          │
│ 持续监听  │   │ 流式识别+实时字幕 │   │  ┌────────────────────┐  │
└──────────┘   └─────────────────┘   │  │ 快路径：规则引擎      │  │
                                     │  │ 正则+同义词表+槽位提取 │  │
                                     │  └────────┬───────────┘  │
                                     │     未命中 ↓              │
                                     │  ┌────────────────────┐  │
                                     │  │ 慢路径：LLM 拆解     │  │
                                     │  │ 复合指令→操作序列    │  │
                                     │  └────────────────────┘  │
                                     └────────────┬─────────────┘
                                                  ↓
               ┌──────────────────┐   ┌──────────────────────────┐
               │     反馈层        │ ← │   绘图 DSL（JSON 指令）   │
               │ TTS 播报          │   └────────────┬─────────────┘
               │ 画布高亮          │                ↓
               │ 操作历史面板      │   ┌──────────────────────────┐
               └──────────────────┘ ← │ 画布执行引擎（Konva.js）   │
                                     │ 绘制/选中/变换/撤销重做     │
                                     └──────────────────────────┘
```

### 2.2 技术栈

| 层 | 选型 | 选型理由 |
|---|---|---|
| 前端框架 | Vue 3 + TypeScript + Vite | 单人项目开发效率高，评委 `npm i && npm run dev` 即可复现；DSL 指令契约用 TS 类型定义，解析器与执行引擎两端获得编译期约束 |
| 画布 | Konva.js | 对象模型天然支持图形的选中/移动/缩放/层级管理，比裸 Canvas 大幅减少状态管理工作量 |
| 语音识别 ASR | 主方案：Web Speech API（`SpeechRecognition`）；备用方案 B：MediaRecorder 录音 + 七牛云 ASR | 主方案浏览器原生、零成本、流式识别延迟最低，**国内网络下需使用 Edge 浏览器**（走 Azure 服务；Chrome 走 Google 服务不可用）。备用方案详见 §6.1 |
| 语音合成 TTS | Web Speech API（`speechSynthesis`） | 浏览器原生，用于操作结果播报与容错追问 |
| 复合指令拆解 | DeepSeek API（JSON 输出模式） | 国内访问稳定、中文理解好、性价比高；输出受 DSL JSON Schema 约束 |
| 状态管理 | Pinia | 管理画布对象列表、操作历史栈、当前选中态 |

**架构形态**：纯前端，无后端。LLM API key 由用户在设置面板中输入，存储于 localStorage，不进入代码仓库。

## 3. 核心模块设计

### 3.1 语音识别层（speech/recognizer.js）

- `SpeechRecognition` 设置 `continuous: true` + `interimResults: true`，实现持续监听与流式字幕。
- **断句策略**：以 `isFinal` 结果为一条指令的边界；用户停顿即触发解析，无需唤醒词（工具属于专注使用场景，全程监听可接受）。
- **自动重连**：Web Speech API 会因静音超时自动停止，`onend` 中自动重启，保证"持续监听"体验。
- **状态指示**：UI 上明确显示「聆听中 / 解析中 / 执行中」三态，让用户知道何时可以说话。

### 3.2 指令解析层（parser/）

#### 双通道路由

```
final 文本 → 文本归一化（去语气词/同义词归一/中文数字转换）
          → 规则引擎匹配
              ├─ 命中且槽位完整 → 直接产出 DSL（~50ms）
              ├─ 命中但槽位缺失 → 使用默认值 或 TTS 追问
              └─ 未命中 / 检测到复合结构（"然后""再""并且"或多动词） → LLM 通道
```

#### 快路径：规则引擎

- **意图识别**：动词关键词表（画/绘/来一个 → draw；删/去掉/移除 → delete；撤销/回退/后退一步 → undo …）。
- **槽位提取**：
  - 图形：`圆|圈|圆形 → circle`，`方|方块|正方形|矩形 → rect` …
  - 颜色：中文颜色词表 → 十六进制（红/大红/红色 → `#e53935`）。
  - 大小：`大一点/小一些/很大` → 相对比例；`半径50` → 绝对值。
  - 位置：九宫格语义（左上/中间/右下…）→ 画布坐标；省略时默认画布中心偏随机偏移（避免重叠）。
- **容错**：
  - 同音/近音字归一（"园/圆"、"花/画"在动词位置的纠正）。
  - 语气词与犹豫词清洗（"嗯""那个""帮我"）。
  - 中文数字转阿拉伯数字（"五十" → 50）。

#### 慢路径：LLM 拆解

- Prompt 中提供：DSL JSON Schema、当前画布对象摘要（id/类型/颜色/位置，供指代消解：“把**那个**圆变大”）、few-shot 示例。
- 输出强制 JSON（操作指令数组），前端用 schema 校验，非法则丢弃并 TTS 告知"没听懂，请换个说法"。
- 复合指令示例：
  - 输入：「画一个房子，左边加一棵树，然后把太阳放到右上角」
  - 输出：`[draw(rect,...), draw(triangle,...), draw(rect,门...), draw(circle,树冠...), draw(rect,树干...), draw(circle,黄色,右上)]`

### 3.3 绘图 DSL（dsl/schema.js）

解析层与执行层之间的唯一契约。示例：

```json
{
  "action": "draw",
  "shape": "circle",
  "props": { "color": "#e53935", "size": "large", "position": "top-left" }
}
```

| action | 说明 | 关键参数 |
|---|---|---|
| `draw` | 绘制图形 | shape, color, size, position |
| `select` | 选中对象 | 按特征（颜色+图形）或指代（"刚才那个"=最近创建） |
| `move` | 移动 | 方向+距离 或 目标位置 |
| `resize` | 缩放 | 相对（大一点）或绝对 |
| `style` | 改属性 | color, fill, stroke |
| `delete` | 删除 | 当前选中 或 按特征 |
| `undo` / `redo` | 撤销/重做 | — |
| `clear` | 清空画布 | 需 TTS 二次确认（破坏性操作） |

设计要点：

- **size / position 支持语义值**（`large` / `top-left`），由执行引擎结合画布尺寸换算为像素，使规则引擎与 LLM 都不需要关心画布分辨率。
- **指代消解（分级实现，P0 刻意收窄）**：
  - **P0 仅支持两类**：`target: "last"`（"刚才那个" = 最近创建/操作的对象）与单特征匹配（颜色 + 图形，"红色的圆"）。匹配到多个时**不猜测**：高亮全部候选并 TTS 追问（"有两个红色的圆，请说'左边的'或'第一个'"）。
  - **P1 再扩展**：空间排序（"左边那个"）、序数（"第二个"）、比较级（"最大的那个"）。这些需要对画布对象做排序与度量比较，规则复杂度高，不在 P0 引入。
- **相对定位（relativeTo）**：LLM 与规则引擎**只输出语义空间关系，禁止输出绝对坐标**——LLM 对坐标数值推理不可靠（会编造 `x:387` 这类无依据的值），且绝对坐标无法校验。结构如下：

  ```json
  {
    "action": "draw",
    "shape": "rect",
    "relativeTo": { "anchors": ["红色的圆", "三角形"], "relation": "between" }
  }
  ```

  `anchors` 是对象特征描述（复用 select 的特征匹配器解析为具体对象），`relation` 为枚举：`left-of / right-of / above / below / between / inside / near`。**坐标计算全部由执行层完成**（结合锚点对象的包围盒与画布尺寸做几何换算），保证 LLM 输出始终可被 schema 校验、可单元测试。anchor 匹配失败时 TTS 反馈"没找到三角形"，不静默落在错误位置。

- **比例坐标（PR #8 修订）**：实现 LLM 通道时发现九宫格粒度不足以摆放组合图形（"画一个人"的头/躯干/四肢需要精确衔接）。在"禁止绝对坐标"原则下放行**0~1 画布比例坐标** `{"fx":0.5,"fy":0.28}`（含直线起止点 from/to）：比例坐标分辨率无关、可校验（越界即拒绝）、对 LLM 没有像素数值的幻觉压力。**像素坐标仍然禁止**。relativeTo（锚定已有对象的相对关系）仍按 P1 计划。

### 3.4 执行引擎与撤销重做（dsl/executor.js, canvas/history.js）

- 命令模式：每条 DSL 指令封装为 `{ apply(), revert() }`，执行后压入历史栈。
- `undo`/`redo` 即栈操作；复合指令（LLM 产出的序列）作为一个事务整体撤销。
- 执行结果回传反馈层：成功 → TTS 简短播报（"已画一个红色的圆"）；失败 → 播报原因。

### 3.5 反馈层

| 通道 | 内容 |
|---|---|
| 实时字幕条 | 流式显示识别中文本（灰）与最终文本（黑），用户随时知道系统"听到了什么" |
| TTS 播报 | 操作结果、错误原因、追问；播报期间暂停识别，避免自听回环 |
| 画布高亮 | 选中对象描边高亮；新绘制对象短暂闪烁定位 |
| 操作历史面板 | 文字列表展示已执行指令，对应撤销层级 |

## 5. 延迟预算

| 阶段 | 目标耗时 |
|---|---|
| 语音 → 流式文字（首字上屏） | < 500ms（Web Speech API 流式） |
| 断句确认（isFinal） | 用户停顿后 ~300-800ms |
| 快路径解析 + 执行 | < 50ms |
| 慢路径（LLM）解析 | 1-2s，期间 UI 显示"思考中"并 TTS 提示 |
| **简单指令端到端** | **目标 < 1.5s** |

## 6. 关键风险与对策

### 6.1 最大风险：Web Speech API 可用性（含备用方案 B）

Web Speech API 是全项目的地基，但其可用性受 **Edge 版本差异、麦克风权限、网络限制** 三重不确定因素影响，存在比赛现场直接不可用的风险。原则：**稳定 > 高级**。

应对策略：

1. **第一周（PR #2 阶段）完成真机验证**：Edge 中文流式识别连续运行 10 分钟以上，覆盖断句、自动重连、权限弹窗流程。
2. **验证失败或不稳定，则启用备用方案 B**：

   ```
   浏览器 MediaRecorder 录音 → 分段上传音频 → 七牛云 ASR → 返回文字 → 进入解析层（完全复用）
   ```

   - ASR 供应商选**七牛云**（主办方服务，国内可用性有保障；DeepSeek 仅有文本模型，不提供 ASR）。
   - 解析层之后的全部链路不变——这正是"解析与执行解耦"带来的可替换性。
   - 代价与已知问题：
     - 端到端延迟由 ~500ms 升至 2-3s（非流式，整段上传）；
     - 断句需自行实现（音量阈值 VAD 或定时切片），无 `isFinal` 事件；
     - 云端 ASR 的 key 不宜暴露在纯前端（签名/CORS 限制），届时可能需补一个极薄的转发代理（单文件 Node 服务），作为独立 PR 提交。
3. 方案 B 在验证通过前**只做设计不写代码**，避免双线投入；ASR 模块接口按可替换设计（`recognizer.js` 对外只暴露 `onResult(text, isFinal)` 回调）。

### 6.2 其他风险

| 风险 | 对策 |
|---|---|
| TTS 播报被麦克风拾取造成自听回环 | 播报期间暂停识别，结束后恢复 |
| LLM 输出非法 JSON | Schema 校验 + 一次重试 + 失败时 TTS 兜底话术 |
| API key 泄露 | key 由用户在设置面板输入，存 localStorage，仓库不含任何密钥 |
| 识别串句（一口气说两条指令） | 连接词切分（"然后/再/接着"）作为规则层预处理 |
