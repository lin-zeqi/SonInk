# 开发记录 #02 — 指令输入层（语音识别 + 调试文本输入 + 实时字幕）

- **日期**：2026-06-12
- **分支**：`feat/02-input`
- **对应路线图**：PR #2（docs/design.md §7）

## 背景与计划调整

原计划本 PR 同步完成 Edge 真机语音验证（design.md §6.1），但当前开发环境**暂无麦克风**。
调整为：

1. Web Speech API 封装照常实现（逻辑薄，风险集中在环境而非代码）；
2. 新增**调试文本输入框**，与语音识别走完全相同的指令提交链路（同一个 `store.submit()`）。
   后续解析/DSL/执行引擎的全部开发与验证均不依赖麦克风；
3. **Edge 真机语音验证顺延**，待有麦克风设备后补做，验证标准不变（中文流式识别连跑
   10 分钟，覆盖断句/自动重连/权限流程）。验证不通过则启用备用方案 B（七牛云 ASR）。

调试输入同时是 demo 演示的兜底通道（仅标注为调试用途，不违背"纯语音控制"的产品定位）。

## 做了什么

1. **`src/speech/recognizer.js`** — Web Speech API 封装：
   - `lang: zh-CN`、`continuous` + `interimResults` 流式识别；
   - 对外契约仅两个回调：`onInterim(text)`（中间结果）/ `onFinal(text)`（一条完整指令），
     备用方案 B 启用时按同一契约替换实现；
   - 静音超时自动重启（`onend` 中检测用户未主动停止则 `start()`），维持持续监听；
   - `no-speech` 错误视为常态不上报；不支持的浏览器返回 `null` 并由 UI 提示换 Edge。
2. **`src/store/command.js`** — 指令流中枢（Pinia）：
   - 语音与调试输入统一汇入 `submit(text, source)`，解析层（PR #4）从这里订阅，
     不感知输入来源；
   - 维护 `interim`（流式字幕）、`history`（指令历史）、`listenState`（聆听状态）。
3. **`src/components/CaptionBar.vue`** — 画布底部实时字幕条：流式中间文本（灰）、
   最近一条最终指令（白，带来源标签）、空闲占位提示。
4. **`src/components/DebugInput.vue`** — 右下角调试输入框，回车提交，橙色"调试输入"标签
   明确区分于正式语音通道。
5. **`src/App.vue`** — 接线：初始化识别器、聆听开关按钮（麦克风权限需要用户手势触发，
   这是保留的唯一一个点击入口；授权开启后即纯语音操作）、浏览器不支持时的警示文案。

## 验证方式

```
npm run dev    # http://localhost:5173
```

- 右下角调试输入框输入"画一个红色的圆"回车 → 底部字幕条显示该文本及"调试"标签；
- 支持语音的浏览器（Edge）：顶栏出现"开始聆听"按钮，点击后变为绿色"聆听中"；
  说话时字幕条流式显示灰色中间文本，停顿后转为白色最终文本（**本条待有麦克风后补验**）；
- 不支持语音的浏览器：顶栏显示橙色提示文案；
- `npm run build` 通过（205 modules）；dev 服务器 HTTP 200。

## 设计决策记录

- **调试输入与语音同链路**：两者都只是 `submit(text, source)` 的调用方，下游完全无感。
  这使"无麦克风"从阻塞项降级为"一条验证待补"。
- **保留一个点击入口（聆听开关）**：浏览器安全策略要求麦克风权限由用户手势触发，
  无法绕过；该按钮仅用于授权/启停，绘图操作本身仍是纯语音。
